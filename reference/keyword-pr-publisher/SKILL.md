# 키워드 기반 PR 발행 스킬

## 목표

1. 키워드로 포스트 초안 생성
2. 자동 PR 생성
3. 리뷰/수정 후 머지
4. Vercel 자동 배포

## 실행 흐름

### Phase 1: 키워드 입력

- GitHub Actions `Create Post PR From Keyword` 실행
- 필수 입력: `keyword`
- 선택 입력: `featured`, `draft`

### Phase 2: 초안 생성

- `scripts/generate-post-from-keyword.mjs`가 `src/content/posts/*.md` 생성
- `OPENAI_API_KEY`가 있으면 AI 생성, 없으면 템플릿 기반 생성

### Phase 3: PR 생성

- `peter-evans/create-pull-request` 액션으로 자동 PR 오픈
- 파일/slug/source 정보가 PR 본문에 자동 포함

### Phase 4: 검토 후 발행

- 리뷰어가 문장/사실/태그 수정
- `main` 머지 시 Vercel 배포

## 검증 규칙

- PR에서 `Validate Blog Build` 워크플로우가 `npm run build` 검증
- draft 게시물은 홈/포스트 목록/상세에서 노출되지 않음

## 기본 작성 원칙 (항상 적용)

이 스킬로 작성하는 포스트는 사용자가 별도로 요청하지 않아도 기본적으로
\"자세하고 풍부한(long-form)\" 품질을 유지한다.

### 품질 벤치마크

- 기준 글: `https://blog.fe4all.dev/posts/claude-code-memory/`
- 스타일 벤치마크: `https://jangkunblog.com/wp/skip-navigation-is-not-a-quick-link/`
- 이후 생성 글은 위 글과 동등한 수준의 구조 밀도와 실행 가능성을 목표로 한다.

### 스타일 레퍼런스 로드 규칙

- 사용자가 \"내 글 스타일\", \"장군블로그 스타일\", \"같은 톤으로\"를 요청하면
  `references/jangkun-writing-pattern.md`를 먼저 읽고 적용한다.
- 접근성/가이드형 포스트 작성 시 위 레퍼런스의 구조 패턴을 기본 템플릿으로 사용한다.

### 풍부함 정의 (운영 기준)

- 구조 밀도: `H2` 9개 이상, `H3` 12개 이상
- 실행 가능성: 단계형 가이드(숫자 목록) + 체크리스트 + 예시(코드/템플릿) 포함
- 실무성: 기능 소개를 넘어 운영 패턴, 리스크, 검증 방법 포함
- 재사용성: 다른 프로젝트에 옮겨 쓸 수 있는 원칙/패턴 제공
- 검토 가능성: 사실/추론 구분, 단정 대신 근거 중심 서술
- 문단 밀도: 단문 나열을 피하고 문단당 3~6문장 중심의 산문형 전개 유지
- 전개 방식: 산문형을 기본으로 하되, 비교/절차/요약/체크포인트는 리스트를 사용해 논리 전달을 우선
- 표준 주제 밀도: `요구사항 -> 동작 원리 -> 입력 규칙 -> 유효/무효 사례 -> 검수` 흐름을 반드시 포함
- 반례 제시: 헷갈리기 쉬운 용어/코드/태그는 \"자주 틀리는 입력\"과 \"권장 입력\"을 짝으로 제시
- 용어 원칙: 한국 실무에서 덜 쓰는 어려운 표현은 쉬운 한국어로 바꿔 쓰고, 필요할 때만 첫 등장에 영어 원어를 괄호로 병기
- 헤딩 규칙: `h1/h2/h3` 제목에는 숫자 넘버링을 사용하지 않음
- 공개성 원칙: 포스트는 특정 팀의 내부 문서가 아니라 누구나 읽을 수 있는 공개 글로 작성
- 주어 원칙: `우리 팀`, `팀이 해야 한다` 같은 특정 집단 주어를 남발하지 않고, 필요할 때는 `우리가` 또는 무주어 문장으로 서술
- 주제 원칙: 키워드에 대한 개인적 정리를 기반으로 기술·사회·인문 관점을 연결해 설명
- 문장 종결 원칙: 본문은 기본적으로 `-다/-이다` 체로 작성하고 `-습니다/-합니다` 체 종결은 사용하지 않음
- 체크리스트 원칙: 체크리스트 항목은 `~했다` 같은 완료형 문장 대신 명사형/행동형(예: `명시`, `병기`, `등록`)으로 간결하게 작성

### 최소 필수 섹션

- 문제 정의
- 배경/맥락
- 표준/근거(날짜 포함)
- 핵심 개념
- 실무 적용 단계
- 안티패턴/리스크
- 개발 검수 방법
- 체크리스트
- 결론

## 기본 응답 정책

- \"짧게 요약\" 요청이 없으면 기본적으로 상세 버전을 작성한다.
- 최신성 영향이 있는 정보는 날짜 기준을 함께 표기한다.
- 사실/추론을 구분해 검토 가능성을 높인다.
- 섹션 제목은 설명형 문장으로 작성하고 번호 접두어를 붙이지 않는다.
- 리스트 사용은 허용이 아니라 권장 대상이 있으며, 독해성이 좋아지는 구간에서는 문단보다 리스트를 우선한다.
- 스펙 설명에서는 반드시 예시(유효/무효 또는 권장/비권장)로 오해 가능성을 선제적으로 제거한다.
