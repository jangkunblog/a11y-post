# Project Agent Rules

## Blog Writing Default

- 블로그 포스트 작성 요청에서는 사용자가 별도로 "짧게"를 요구하지 않는 한 기본적으로 상세(long-form) 버전으로 작성한다.
- 기본 구조는 다음 순서를 따른다: 문제 정의 -> 배경/맥락 -> 핵심 개념 -> 실무 적용 단계 -> 자주 하는 실수와 주의점 -> 체크리스트 -> 마무리.
- 내용은 항상 실행 가능해야 하며, 체크리스트/단계/예시 중 최소 1개 이상을 반드시 포함한다.
- 과장된 홍보 문구 대신 근거 중심의 실무 문체를 유지한다.
- 품질 벤치마크는 `https://blog.fe4all.dev/posts/claude-code-memory/` 수준으로 유지한다.
- 기본 분량/구조 기준: H2 9개 이상, H3 12개 이상, 단계형 가이드와 체크리스트를 반드시 포함한다.
- 스타일 벤치마크는 `https://jangkunblog.com/wp/skip-navigation-is-not-a-quick-link/`로 유지한다.
- 문단 톤/전개 기준: 오해 바로잡기 -> 개념 정의 -> 표준 근거 -> 올바른 사용법 -> 잘못된 사례 -> 검수 방법 -> 마무리 순서를 기본으로 한다.
- 용어 표기 원칙: `안티패턴`은 `❌ 잘못된 사례`, `대안 패턴`은 `🔵 권장하는 사례`로 표기한다.
- 이모지 사용 원칙: `❌`, `🔵` 이모지는 섹션 제목(헤딩)에서만 사용하고 본문 문장에서는 사용하지 않는다.
- 한 문단은 3~6문장을 기본으로 하며, 섹션마다 설명 문단을 충분히 확보해 단문 나열형 글이 되지 않도록 한다.
- 한 문장 문단은 강조가 필요한 경우에만 제한적으로 사용한다.
- 산문형을 기본으로 하되, 비교/요약/절차/체크 항목처럼 논리 전달이 더 명확해지는 구간은 리스트(불릿/번호)를 우선 사용한다.
- 표준/스펙 주제는 정의 요약에서 끝내지 않고 `요구사항 -> 동작 원리 -> 입력 규칙 -> 유효/무효 사례 -> 검수 방법` 순서로 설명한다.
- 용어가 헷갈리기 쉬운 항목(예: 코드 체계/태그 형식)은 반례와 교정 예시를 함께 제시한다.
- 용어 원칙: 한국 실무에서 덜 쓰는 어려운 표현은 쉬운 한국어로 바꿔 쓰고, 필요할 때만 첫 등장에 영어 원어를 괄호로 병기한다.
- 공개성 원칙: 모든 글은 특정 팀의 내부 가이드가 아니라 누구나 읽을 수 있는 공개 포스팅으로 작성한다.
- 주어/시점 원칙: `우리 팀`, `팀이 해야 한다` 같은 특정 집단 주어를 남발하지 않고, 필요할 때는 `우리가` 또는 무주어 문장으로 서술한다.
- 글의 목적 원칙: 글은 키워드에 대한 개인적 정리를 공개적으로 공유하는 형식을 유지하며, 기술·사회·인문 주제를 실제 맥락에 연결해 설명한다.
- 문장 종결 원칙: 본문 문장은 기본적으로 `-다/-이다` 체로 작성하고, `-습니다/-합니다` 체 종결은 사용하지 않는다.
- 체크리스트 원칙: 체크리스트 항목은 `~했다` 같은 완료형 문장 대신 명사형/행동형(예: `명시`, `병기`, `등록`)으로 간결하게 작성한다.
- `h1`, `h2`, `h3` 제목에는 숫자 넘버링(`1.`, `1)`, `01.` 등)을 붙이지 않는다.
- 순서가 필요한 내용은 제목이 아니라 본문의 번호 목록/체크리스트로 표현한다.
