제품을 설계하고 구현할 때 `lang` 속성은 자주 "기본 속성 하나" 정도로 취급됩니다. 그런데 이 항목은 스크린리더 발음 규칙·점자 변환·번역·폰트 선택·문장 처리 방식 전체를 좌우하는 기반 정보입니다. 문서의 기본 언어가 비어 있거나 잘못 지정되면, 보조기기는 가장 먼저 잘못된 가정 위에서 동작하게 됩니다. 그리고 이 오차는 페이지 전체에 누적됩니다.

특히 다국어 콘텐츠를 운영하는 팀에서는 "언어코드와 지역 하위태그를 어디까지 붙여야 하는가"에서 혼선이 반복됩니다. `ko-KR`만 써야 하는지, `ko`면 충분한지, `KR`만 쓰면 되는지, `xml:lang`을 같이 써야 하는지 같은 질문이 계속 나옵니다. 이 글은 그런 반복 질문을 끊기 위해, WCAG와 HTML 스펙, 그리고 BCP 47 규칙을 한 문서에서 연결해 정리합니다.

## 기본 언어 제공이 왜 접근성 핵심인지

WCAG의 읽기 가능성(Readable) 맥락에서 언어 정보는 보조적 힌트가 아니라 해석의 기준점입니다. W3C Understanding 문서는 페이지 언어가 명시되면 스크린리더가 올바른 발음 규칙을 적용하고, 시각 브라우저가 문자·스크립트를 더 정확히 처리하며, 미디어 플레이어가 자막 처리에도 도움을 받을 수 있다고 설명합니다. 즉, 언어 정보는 "어떻게 읽을지"에 대한 엔진의 입력값입니다.

실무에서는 이 항목이 종종 "다국어 사이트에서만 중요하다"고 오해됩니다. 하지만 단일 언어 사이트라도 기본 언어 선언이 빠지면 사용자 에이전트는 추정에 의존하게 되고, 추정 실패 시 발음·억양·문맥 처리가 어긋납니다. 다시 말해 언어 선언은 다국어 기능이 있을 때 추가하는 옵션이 아니라, 모든 페이지가 기본으로 가져야 할 문서 메타데이터입니다.

## WCAG에서 요구하는 수준

WCAG 2.2 본문에서 기본 언어와 부분 언어는 별도 성공 기준으로 분리됩니다. 이 분리는 매우 중요합니다. 페이지 전체의 기본 언어만 맞춰도 Level A 일부는 충족되지만, 문장/구절 단위 언어 변경까지 정확히 표시해야 실제 사용 경험이 안정됩니다.

### 페이지 기본 언어 요건

WCAG의 Language of Page 기준은 "각 웹 페이지의 기본 인간 언어를 프로그래밍 방식으로 판별 가능해야 한다"는 요구입니다. 여기서 핵심은 "기본(default)"이라는 단어입니다. 페이지 안에 여러 언어가 섞여 있어도, 우선순위 언어를 하나 정해 기계가 식별할 수 있어야 한다는 뜻입니다. WCAG Understanding 문서도 다국어 페이지에서 기본 언어는 가장 많이 쓰인 언어를 기준으로 정하라고 설명합니다.

### 페이지 내부 언어 변경 요건

Language of Parts 기준은 구절·문장 단위의 언어 전환까지 식별 가능해야 한다는 요구입니다. 다만 예외도 명시돼 있습니다. 고유명사, 기술용어, 언어 판별이 불가능한 단어, 주변 텍스트에 동화된 차용어는 예외로 볼 수 있습니다. 이 예외 규칙 때문에 실무에서는 "모든 외래어를 무조건 `lang` 처리"하는 기계적 접근보다, 실제 발화/해석 변화가 생기는 지점을 구분하는 판단이 필요합니다.

### WCAG Technique가 제시하는 구현 포인트

W3C Technique H57과 H58은 다음을 실무 체크로 제시합니다.

- 페이지 기본 언어는 `html` 요소의 `lang`으로 선언
- 페이지 내 언어 전환은 해당 요소에 `lang`을 추가해 구분
- `lang` 값은 BCP 47(또는 그 후속 규격) 형식을 따름

즉, WCAG 관점의 최소 구현 원칙은 "루트에서 기본 언어 선언 + 변경 구간만 국소적으로 오버라이드"로 요약할 수 있습니다.

## HTML 스펙에서 lang가 실제로 동작하는 방식

HTML Living Standard는 `lang`을 전역 속성으로 정의하고, 값은 "유효한 BCP 47 언어 태그 또는 빈 문자열"이어야 한다고 명시합니다. 그리고 빈 문자열(`lang=""`)은 "기본 언어를 알 수 없음(unknown)"을 의미합니다. 이 규정은 언어값이 없을 때의 처리까지 표준화해 두었다는 점에서 중요합니다.

또한 스펙은 상속 규칙을 명확히 둡니다. 어떤 요소에 `lang`이 없으면 부모의 언어를 상속합니다. 따라서 루트 `html`의 기본값 하나가 문서 전체의 디폴트가 되고, 필요한 지점에서만 하위 요소에 다른 `lang`을 지정하면 됩니다. 이 상속 모델을 이해하면 과도한 중복 마크업을 줄일 수 있습니다.

`xml:lang`과의 관계도 자주 오해되는 부분입니다. HTML 스펙은 HTML 문서에서 XML 네임스페이스의 `lang` 속성 사용을 권장하지 않으며, HTML에서는 기본적으로 `lang` 속성을 쓰도록 안내합니다. XML 문서 맥락이나 호환 문서에서 `xml:lang`이 등장할 수는 있지만, 현대 HTML 운영에서 기본 해법은 `lang` 단일 사용입니다.

## BCP 47과 ISO 코드 관계를 정확히 이해하기

사용자들이 자주 "ISO BCP"라고 묶어 부르지만, 엄밀히 말하면 BCP 47은 ISO 표준명이 아니라 IETF의 Best Current Practice입니다. W3C 국제화 문서도 BCP를 "RFC 묶음의 지속 이름"으로 설명하며, 최신 문법 기준 문서로 RFC 5646을 가리킵니다. 또한 BCP 47은 RFC 5646(태그 구조)와 RFC 4647(태그 매칭)으로 구성된다는 점을 IETF Datatracker가 명시합니다.

다만 BCP 47의 하위 코드 체계는 ISO 계열 코드와 깊게 연결됩니다. W3C의 "Language tags in HTML and XML" 문서는 다음 관계를 분명히 보여줍니다.

- 언어 하위태그: ISO 639 계열
- 스크립트 하위태그: ISO 15924
- 지역 하위태그: ISO 3166 2자리 + UN M.49 3자리
- 실제 사용 가능한 값의 단일 출처: IANA Language Subtag Registry

즉, 실무적으로는 "ISO 목록을 직접 조합"하기보다 IANA 레지스트리를 기준으로 태그를 확정하는 방식이 가장 안전합니다.

### 태그 형식을 먼저 보면 헷갈림이 줄어든다

BCP 47은 단순히 `언어-지역(Region)` 2단 구조만을 의미하지 않습니다. 실무에서 자주 쓰는 축약 패턴은 아래처럼 이해하면 됩니다.

```text
language[-script][-region][-variant...]
```

- `language`는 필수이며 첫 하위태그에 위치합니다.
- `script`, `region`, `variant`는 필요할 때만 붙는 선택 하위태그입니다.
- 하위태그 구분자는 하이픈(`-`)입니다. 언더스코어(`_`)는 표준 형식이 아닙니다.

즉 "`ISO(언어)-ISO(지역)`만 알면 충분하다"가 아니라, 그 조합은 BCP 47의 여러 합법 패턴 중 하나일 뿐입니다. 중국어처럼 스크립트 구분이 중요한 언어에서는 `language-script-region` 형태가 오히려 더 정확할 수 있습니다.

### 대소문자와 하이픈 표기에서 자주 생기는 오해

BCP 47 값은 비교 시 대소문자를 구분하지 않습니다. 하지만 실무 가독성과 관례를 위해 보통 다음처럼 씁니다.

- 언어: 소문자 (`ko`, `en`, `zh`)
- 스크립트: 첫 글자 대문자 (`Latn`, `Hans`, `Hant`)
- 지역: 대문자 (`KR`, `US`, `GB`) 또는 숫자 (`419`)

예를 들어 `ko-kr`은 해석 자체는 가능하지만, 보통 `ko-KR`처럼 정규화된 표기를 권장합니다. 가독성, 코드리뷰, 자동화 도구의 일관성 측면에서 정규 표기가 유리합니다.

### 유효/무효와 오해 사례를 함께 보는 것이 가장 빠르다

- `ko` : 유효. 한국어 일반 선언의 기본값으로 충분한 경우가 많음
- `ko-KR` : 유효. 한국어 + 지역(대한민국) 구분이 필요할 때 사용
- `zh-Hans-CN` : 유효. 언어 + 스크립트 + 지역을 모두 표현
- `es-419` : 유효. 지역에 UN M.49 숫자 코드(라틴아메리카)를 사용한 사례
- `KO-kr` : 해석은 가능하지만 비권장 표기. 일반적으로 `ko-KR`로 정규화
- `KR` : 기본 언어 태그로는 부적절. 첫 하위태그가 언어여야 하기 때문
- `ko_kr` : 형식 오류. 하위태그 구분자는 하이픈(`-`)이어야 함
- `kr-KR` : 형식상 유효할 수 있으나, 의도가 "한국어"라면 잘못된 값일 가능성이 큼. IANA 레지스트리 기준 `kr`는 Korean이 아니라 Kanuri이며, 한국어는 `ko`

## 지역 하위태그 입력 규칙에서 가장 많이 틀리는 부분

지역 하위태그 입력에서 반복되는 오류는 거의 동일한 패턴으로 나타납니다. 핵심은 언어 태그가 항상 "언어"로 시작해야 한다는 점입니다. W3C "Choosing a Language Tag" 문서도 첫 하위태그는 기본적으로 언어(Primary language subtag)라고 명확히 설명합니다.

### 반드시 지켜야 할 규칙

- 언어 태그의 첫 하위태그는 언어 코드여야 함
- 지역 코드는 언어 뒤에 붙는 선택 하위태그임
- 지역 코드는 필요할 때만 추가하고, 불필요하면 더 짧은 태그를 사용

### 입력 예시로 보는 판단 기준

- `ko` : 한국어 콘텐츠 일반 선언에 충분한 기본값
- `ko-KR` : "대한민국에서 쓰이는 한국어" 구분이 실제로 필요할 때 사용
- `en-GB`, `en-US` : 영어 변형 구분이 필요한 경우
- `zh-Hans`, `zh-Hant` : 중국어를 문자체계 기준으로 구분할 때 유용
- `KR` : 언어 하위태그 없이 지역만 있는 형태라 기본 언어 태그로 부적절

여기서 중요한 포인트는 "길게 쓰는 것이 더 정확한 것"이 아니라는 점입니다. W3C 국제화 가이드는 언어 태그를 가능한 짧게 유지하고, 구분이 필요할 때만 하위태그를 추가하라고 권장합니다. 따라서 모든 페이지를 습관적으로 `xx-YY` 형태로 강제하는 정책은 오히려 유지보수성을 떨어뜨릴 수 있습니다.

## 구현 패턴

언어 선언 구현은 복잡할 필요가 없습니다. 기본값을 루트에 선언하고, 전환 구간만 국소적으로 마킹하면 대부분의 요구사항을 안정적으로 충족할 수 있습니다.

### 페이지 기본 언어 선언

```html
<!doctype html>
<html lang="ko">
  <head>
    <meta charset="utf-8" />
    <title>예시 문서</title>
  </head>
  <body>
    <h1>기본 언어가 한국어인 페이지</h1>
  </body>
</html>
```

### 문장 일부 언어 전환 마킹

```html
<p>
  릴리스 노트에서
  <span lang="en">Breaking Change</span>
  구간은 별도 요약으로 제공됩니다.
</p>
```

### 인용문/블록 단위 언어 전환 마킹

```html
<blockquote lang="de">
  Da dachte der Herr daran, ihn aus dem Futter zu schaffen ...
</blockquote>
```

## 자주 하는 실수와 주의점
표준을 알고도 운영 과정에서 무너지는 지점이 있습니다. 아래 항목은 실제 프로젝트에서 반복적으로 발견되는 오류 패턴입니다.

- `meta http-equiv="Content-Language"`만으로 기본 언어를 해결하려는 시도
- 기본 언어를 `body`에만 선언하고 `html` 루트 선언을 비워두는 방식
- 다국어 구절을 전혀 마킹하지 않아 TTS 발음이 문장 중간에 깨지는 현상
- 지역코드를 언어코드처럼 오인해 첫 하위태그 자리에 입력하는 실수
- 구분 필요가 없는데도 모든 페이지를 과도하게 `언어-지역` 형태로 고정

W3C 국제화 가이드는 텍스트 처리 관점의 기본 언어 선언은 `html`의 언어 속성으로 처리하고, `Content-Language` 메타는 그 목적의 대체 수단이 아니라고 분리해서 안내합니다. 이 구분을 팀 규칙으로 명시해 두면 재발을 크게 줄일 수 있습니다.

## 개발·디자인·기획이 함께 보는 점검 포인트

이 문서는 심사자만 보는 체크리스트가 아니라, 기능을 설계하고 만드는 팀 전체를 위한 기준 문서입니다. 그래서 점검 항목도 "누가 잡아내느냐"보다 "어떤 단계에서 무엇을 확인하면 재작업이 줄어드는가"에 초점을 맞추는 편이 좋습니다. 핵심은 자동과 수동을 역할 분담해 쓰는 것입니다. 자동은 형식 오류를 빠르게 제거하고, 수동은 실제 읽기 경험이 자연스러운지를 확인합니다.

### 자동 점검으로 먼저 정리할 항목

자동 점검은 개발 단계에서 실수를 초기에 줄여 줍니다. 기획/디자인이 만든 화면 구조가 실제 마크업으로 내려왔을 때 기본 규칙이 깨지지 않았는지 빠르게 확인할 수 있습니다.

- 루트 `html` 요소의 `lang` 존재 여부
- `lang` 값의 BCP 47 형식 유효성(하이픈 사용, 첫 하위태그 언어 여부)
- `ko_kr`, `KR` 같은 명백한 오입력 패턴 탐지
- 페이지 내 `lang` 속성 오타/중복/빈 문자열(`lang=""`) 사용 위치 점검
- 템플릿 단위 누락(특정 레이아웃에서만 `lang` 빠짐) 여부

권장 적용 시점은 다음과 같습니다.

- PR 단계: 마크업 린트 + 정적 HTML 검사
- 빌드 단계: 대표 URL 샘플 점검
- 정기 점검: 크롤링 기반 전수 확인

### 수동 확인이 필요한 항목

수동 확인은 "형식은 맞는데 사용자 경험은 어색한" 문제를 찾는 단계입니다. 특히 다국어가 섞인 문장, UI 라벨, 동적 컴포넌트는 사람이 맥락을 보고 판단해야 품질이 올라갑니다.

- 화면리더(TTS)로 문장 중간 언어 전환 구간 발음/억양 확인
- 언어 전환 마킹이 필요한 구간에만 적용됐는지 확인(과도 마킹 방지)
- 기본 언어가 실제 본문 비중과 일치하는지 확인
- 번역되지 않은 UI 조각(버튼/배지/라벨)의 언어 표시 누락 확인
- 모바일/데스크톱/웹뷰 등 주요 환경에서 발화 품질 차이 확인

여기서 디자이너는 문구/레이블 일관성을, 기획자는 언어 정책과 콘텐츠 흐름을, 개발자는 마크업 구현과 상속/오버라이드 처리를 함께 확인하면 가장 효율적입니다. 즉 `lang`은 특정 직군의 단독 과제가 아니라, 텍스트를 다루는 모든 역할의 공통 품질 항목입니다.

## 실무 체크리스트

- [ ] 모든 페이지의 `html` 요소에 기본 `lang`이 선언되어 있다.
- [ ] 페이지 내부 언어 전환 구간은 요소 단위 `lang`으로 구분되어 있다.
- [ ] `lang` 값은 BCP 47 기준으로 검증된다.
- [ ] 지역 하위태그만 단독으로 언어 태그에 사용하지 않는다.
- [ ] 필요 없는 지역 하위태그를 습관적으로 붙이지 않는다.
- [ ] `meta Content-Language`를 텍스트 처리용 기본 언어 선언 대체로 사용하지 않는다.
- [ ] 자동 점검(형식/누락)과 수동 확인(발화/맥락)이 역할 분담되어 운영된다.
- [ ] 기능 릴리스 전에 대표 시나리오에서 언어 전환 구간이 실제로 검토된다.

## 마무리

기본 언어 제공은 접근성에서 가장 단순해 보이지만, 실제로는 전체 텍스트 처리 품질을 결정하는 기초 계약입니다. WCAG는 페이지 기본 언어와 부분 언어 전환을 분리해 요구하고, HTML 스펙은 `lang`의 유효값·상속·unknown 상태를 명확히 정의합니다. 그리고 BCP 47은 ISO 코드들을 실무에서 일관되게 쓰도록 연결해 주는 언어 태그 체계입니다.

따라서 운영 원칙도 단순하게 가져가면 됩니다. 루트에 기본 언어를 선언하고, 필요한 지점만 언어 전환을 마킹하며, 지역코드는 구분이 필요할 때만 붙입니다. 이 세 가지만 지켜도 접근성 품질과 국제화 품질이 동시에 안정됩니다.
